1. ¿Qué hemos construido?

Hemos creado la web oficial de la AFA Remontival, la asociación de familias de alumnos del colegio Remontival de Pamplona. La web tiene dos partes:

  • Parte pública – visible para cualquier persona que entre. Muestra noticias, actividades, consejos para familias, menús del comedor y cómo contactar.
  • Parte privada – solo para socios. Permite identificarse con email y contraseña para acceder a contenido o funciones exclusivas.

La web está disponible en dos idiomas: castellano y euskera, y se puede visitar en:
https://robeche.github.io/afa/

Concepto clave: web estática vs. dinámica.
Una web dinámica genera cada página en tiempo real cuando alguien la visita (como lo hacía la web anterior en Django/Python). Una web estática genera todos los archivos HTML de antemano y los sirve tal cual, como si fueran documentos. Esto la hace muy rápida y barata de hospedar. La nueva web de la AFA es estática, aunque se conecta a una base de datos para mostrar datos actualizados.

2. Las herramientas que usamos

Para construir la web hemos utilizado varias herramientas, cada una con un propósito concreto:

Next.js

El marco principal que estructura toda la web. Es como el "esqueleto" sobre el que construimos todo.

TypeScript

El lenguaje de programación que usamos. Es una versión mejorada de JavaScript que ayuda a detectar errores antes de que ocurran.

Tailwind CSS

Sistema de estilos visuales. Define colores, tamaños y disposición de los elementos en pantalla.

Supabase

La base de datos y el sistema de login en la nube. Guarda noticias, actividades, socios, etc.

GitHub

Almacén del código fuente. Guarda todos los archivos del proyecto con historial de cambios.

GitHub Pages

El servidor gratuito donde está publicada la web. Cada vez que se sube código nuevo, la web se actualiza sola.

Bootstrap Icons

Colección de iconos listos para usar (casitas, sobres, calendarios…) que hacen la web más visual.

VS Code

El editor de texto especializado donde se escribe el código, con ayuda inteligente de GitHub Copilot.

¿Por qué Next.js y no WordPress?
WordPress es muy popular pero necesita un servidor que esté ejecutándose constantemente. Next.js puede generar una web estática que funciona en servidores gratuitos como GitHub Pages, sin coste mensual de hosting. Además es mucho más rápida y segura.

3. Cómo se organiza el proyecto

Un proyecto web moderno no es un solo archivo, sino una carpeta con cientos de archivos organizados. Aquí está la estructura principal:

📁 WebAFA/
  📁 src/  ← todo el código fuente está aquí
    📁 app/  ← cada carpeta = una URL de la web
      📁 (public)/[lang]/  ← páginas públicas en /es o /eu
        📄 page.tsx  ← portada
        📁 noticias/  ← lista de noticias y detalle
        📁 actividades/
        📁 consejos/
        📁 comedor/
        📁 contacto/
      📁 (private)/socios/  ← área privada de socios
    📁 components/  ← piezas reutilizables de la interfaz
      📁 layout/  ← cabecera, pie, envoltorio general
      📁 ui/  ← tarjetas, carrusel, selector de idioma
    📁 lib/  ← lógica y utilidades
      📁 i18n/  ← textos en castellano y euskera
      📁 supabase/  ← conexión a la base de datos
    📁 types/  ← definición de qué datos tiene cada cosa
  📁 public/  ← imágenes, logos y archivos estáticos
    📁 assets/logos/  ← LogoHorizontal.png
  📁 supabase/migrations/  ← instrucciones SQL para crear la base de datos
  📁 .github/workflows/  ← automatización del deploy
  📄 next.config.ts  ← configuración principal de Next.js
  📄 .env.local  ← contraseñas y claves secretas (no en GitHub)
¿Qué es un componente?
Un componente es una pieza de interfaz reutilizable. Por ejemplo, la tarjeta de noticia (imagen + título + fecha + resumen) es un componente. Se define una sola vez y se usa en todas las páginas que lo necesitan. Si cambias el diseño de la tarjeta, el cambio se aplica en todos los sitios automáticamente.

4. Las páginas de la web

En Next.js, cada carpeta dentro de app/ es una página. La URL que ve el usuario en el navegador corresponde directamente a la estructura de carpetas.

URL en el navegador Qué muestra
/es o /eu Portada principal con noticias y llamadas a acción
/es/noticias Lista de todas las noticias
/es/noticias/fiesta-fin-curso Noticia concreta (el texto después de noticias/ es el "slug")
/es/actividades Próximas actividades de la AFA
/es/comedor PDFs del menú del comedor escolar
/es/consejos Consejos y orientación para familias
/es/contacto Dirección y datos de contacto
/socios Área privada: formulario de login
¿Qué es un "slug"?
Un slug es el identificador único de un contenido en la URL. Por ejemplo, la noticia sobre la fiesta de fin de curso tiene el slug fiesta-fin-curso. Así la URL es legible y descriptiva en lugar de mostrar números crípticos como /noticia?id=47.

5. Dos idiomas: castellano y euskera

Toda la web está disponible en los dos idiomas. Esto se llama internacionalización (abreviado i18n). Así es como funciona:

El idioma en la URL

El segmento [lang] en la URL es dinámico: puede ser es o eu. Next.js genera páginas distintas para cada idioma cuando construye el proyecto.

Los textos en archivos JSON

En lugar de escribir los textos directamente en el código, los guardamos en archivos de traducción:

// src/lib/i18n/es.json (castellano) { "siteName": "AFA Remontival", "menu": { "noticias": "Noticias", "comedor": "Comedor" } } // src/lib/i18n/eu.json (euskera) { "siteName": "AFA Remontival", "menu": { "noticias": "Berriak", "comedor": "Jantokia" } }

El código pregunta: "¿en qué idioma estamos?" y carga el archivo correspondiente. El contenido de las noticias también se guarda en dos versiones (titulo_es / titulo_eu) en la base de datos.

6. La base de datos (Supabase)

Una base de datos es como una hoja de cálculo en la nube, pero mucho más potente y segura. Usamos Supabase, un servicio gratuito basado en PostgreSQL.

Las tablas que hemos creado

Cada tabla guarda un tipo de información diferente:

  • noticias – título, resumen, contenido, fecha, imagen, publicada (sí/no)
  • actividades – título, descripción, fecha inicio, fecha fin, ubicación
  • consejos – título, contenido, imagen, orden de aparición
  • documentos_comedor – mes, año, enlace al PDF en castellano, enlace en euskera
  • socios_perfiles – nombre, apellidos, rol (admin / editor / socio)
  • contacto_mensajes – mensajes enviados desde el formulario de contacto

La seguridad: Row Level Security (RLS)

RLS es un sistema de permisos que actúa directamente en la base de datos. Define quién puede ver o modificar cada fila de cada tabla. Hemos configurado estas reglas:

  • Cualquier visitante puede leer noticias, actividades y consejos marcados como publicados.
  • Solo admin y editor pueden crear, editar o borrar contenido.
  • Cada socio puede ver solo su propio perfil.
  • El formulario de contacto puede enviar mensajes sin login, pero solo admin/editor puede leerlos.
¿Por qué es importante RLS?
En muchas webs, la seguridad se programa solo en el servidor web. Si alguien encuentra un fallo, puede acceder a todos los datos. Con RLS, aunque alguien saltara el servidor, la propia base de datos rechazaría las peticiones no autorizadas. Es una capa extra de protección.

El archivo de migración SQL

Toda la estructura de la base de datos está escrita en un archivo llamado 0001_init.sql. Ejecutarlo en Supabase crea todas las tablas, permisos y funciones de una sola vez. Esto significa que cualquier persona con acceso al proyecto puede reproducir exactamente la misma base de datos.

7. El acceso privado para socios

La página /socios tiene un formulario de email y contraseña. Supabase Auth se encarga de todo el proceso de autenticación de forma segura:

  • El socio escribe su email y contraseña El formulario envía los datos a Supabase Auth a través de internet cifrado (HTTPS).
  • Supabase verifica las credenciales Comprueba en su base de datos segura si el email y contraseña coinciden.
  • Supabase devuelve un token de sesión Un token es como un pase temporal que dice "este usuario ha iniciado sesión correctamente". Se guarda en el navegador.
  • La web muestra el área privada Con el token activo, el usuario ve su nombre y el botón de "Cerrar sesión". El token caduca automáticamente.

Las contraseñas nunca se guardan en texto plano; Supabase las almacena como un hash (una huella digital irreversible).

8. Cómo se publica en internet automáticamente

Cada vez que se sube código nuevo a GitHub, un proceso automático construye y publica la web sin intervención manual. Esto se llama CI/CD (Integración y Despliegue Continuos).

  • Se sube código a GitHub El desarrollador ejecuta git push desde su ordenador. Los archivos llegan al repositorio en GitHub.
  • GitHub Actions detecta el cambio Un archivo de configuración (deploy-pages.yml) le dice a GitHub qué hacer cuando llega código nuevo.
  • Se construye la web en un servidor de GitHub Next.js compila todo el código y genera archivos HTML, CSS y JS listos para servir. Este proceso dura unos 40 segundos.
  • Los archivos se suben a GitHub Pages GitHub Pages sirve esos archivos estáticos como una web pública. Es completamente gratuito.
  • La web está actualizada En menos de 2 minutos desde el push, los visitantes ven la nueva versión.
¿Qué son los "secrets" de GitHub?
Las claves privadas de Supabase (URLs y contraseñas) no pueden estar en el código porque GitHub es público. Se guardan como "secrets" en la configuración del repositorio de GitHub. El proceso de construcción las inyecta en el momento de compilar, sin que nadie pueda verlas.

9. El flujo completo de trabajo

Este diagrama resume cómo encajan todas las piezas:

Desarrollo → VS Code + GitHub Copilot → se escribe y prueba el código en local
Control de versiones → git push → el código va a GitHub con historial completo
Automatización → GitHub Actions compila y despliega automáticamente en GitHub Pages
Usuarios → visitan la web en su navegador → ven HTML estático muy rápido
Datos en vivo → Supabase sirve noticias, hace el login y gestiona socios en tiempo real
